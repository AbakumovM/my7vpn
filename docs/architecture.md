# Архитектура VPN-бота — главный справочник

**Статус:** актуально
**Последнее обновление:** 2026-04-25

---

## Суть проекта

Telegram VPN-бот для продажи подписок на VPN-сервис. Пользователь покупает подписку через бота, бот регистрирует его в панели Remnawave и отправляет ссылку для подключения в happ.

**Ключевая идея:** одна подписка на пользователя, а не на устройство. Количество устройств (1/2/3) указывается при покупке и контролируется панелью Remnawave через `hwid_device_limit` — **наш код устройствами не управляет**.

---

## Бизнес-модель

### Что покупает пользователь

1. **Количество устройств** — 1, 2 или 3 (выбирает при покупке)
2. **Срок подписки** — 1, 3, 6 или 12 месяцев
3. Получает единый **subscription_url** для happ — он работает на всех выбранных устройствах

### Тарифная матрица (TARIFF_MATRIX)

| Устройств | 1 мес | 3 мес | 6 мес | 12 мес |
|-----------|-------|-------|-------|--------|
| 1         | 150₽  | 400₽  | 700₽  | 1200₽  |
| 2         | 250₽  | 650₽  | 1100₽ | 1900₽  |
| 3         | 350₽  | 900₽  | 1500₽ | 2600₽  |

### Оплата

Два режима, управляются флагом `yookassa.enabled` в конфиге:

- **`enabled=false`** (текущий режим): QR-код → «Я оплатил» → бот создаёт PendingPayment → администратор подтверждает вручную → бот создаёт аккаунт в Remnawave
- **`enabled=true`** (автоматический): ЮKassa → webhook → автоподтверждение, PendingPayment → Remnawave

### Реферальная программа

- Реферал приводит друга → другу +7 дней бесплатно, рефералу +50₽ на баланс
- Баланс списывается при следующей оплате автоматически

## Модель данных

### Что хранится в нашей БД

```
User
├── telegram_id       — уникальный Telegram ID
├── remnawave_uuid    — UUID пользователя в панели Remnawave (NULL у старых)
├── subscription_url  — ссылка для happ (от Remnawave)
├── balance           — бонусный баланс в рублях
├── email             — для чеков ЮKassa и сайта
├── referral_code     — уникальный код для реферальных ссылок
├── referred_by       — telegram_id того, кто пригласил
└── free_months       — использовал ли бесплатный период

Subscription (привязана к User напрямую, НЕ к Device)
├── user_id           — FK → users.id
├── start_date
├── end_date
├── device_limit      — сколько устройств было в подписке
├── plan              — срок в месяцах
└── is_active

Payment (история платежей — никогда не удаляется)
├── user_telegram_id  — денормализован для надёжности истории
├── subscription_id   — FK → subscriptions.id
├── amount            — сумма в рублях
├── currency          — RUB
├── payment_method    — "карта", "yookassa", etc.
├── payment_date
├── duration          — срок за который оплачено (месяцев)
├── device_limit      — снимок: сколько устройств было
├── status            — "success" | "pending" | "failed"
└── external_id       — ID платежа в ЮKassa (для сверки)

PendingPayment (временная, удаляется после подтверждения)
├── user_telegram_id
├── action            — "new" | "renew"
├── duration
├── amount
├── balance_to_deduct
├── device_limit
└── created_at
```

### Что НЕ хранится в новой модели

**Таблица `devices`** — устаревшая, от старой device-based модели. В новом коде не создаётся. Хранится в БД только для истории старых пользователей.

Причина: Remnawave управляет устройствами через `hwid_device_limit` на уровне пользователя — нам нечего трекать на своей стороне.

## Remnawave — правила работы

### Принцип: 1 telegram_id = 1 Remnawave User

- `create_user(telegram_id, expire_at, device_limit)` — только при первой покупке / миграции
- `update_user(uuid, expire_at, device_limit)` — при каждом продлении
- Определение нового/старого: `user.remnawave_uuid is None` → `create_user`, иначе → `update_user`

### Что панель делает сама

- Контролирует количество подключённых устройств (`hwid_device_limit`)
- Отключает пользователя по `expire_at`
- Возвращает единый `subscription_url` — не меняется при продлении

### Формат username в Remnawave

`tg{telegram_id}` — гарантированно уникально.

## Флоу покупки — 3 шага

```
Шаг 1: Выбор количества устройств (1 / 2 / 3)
    ↓
Шаг 2: Выбор тарифа (1 / 3 / 6 / 12 мес) — цены из TARIFF_MATRIX
    ↓
Шаг 3: Подтверждение (итого с учётом бонусного баланса)
    ↓
[Оплата: QR-код или ЮKassa]
    ↓
PendingPayment создаётся в БД
    ↓
Администратор подтверждает (или ЮKassa webhook)
    ↓
confirm_payment:
  ├── Создать Subscription + Payment в нашей БД
  ├── Если remnawave_uuid = NULL → create_user (новый или миграция)
  └── Если remnawave_uuid задан → update_user (продление)
    ↓
Пользователь получает subscription_url
```

### Флоу продления (renew)

Идентичен флоу покупки, но `action="renew"`.
При `action="renew"` в `confirm_payment`:
- Находим активную Subscription пользователя по `user_telegram_id`
- Продлеваем `end_date` от текущего конца (или от now, если уже истекла)
- Вызываем `update_user` в Remnawave с новым `expire_at` и `device_limit`
- **Не используем device_name** — в новой модели его нет

## Миграция старых пользователей

### Кто такие «старые» пользователи

`users.remnawave_uuid IS NULL` — работали на старой системе (3x-ui / device-based модель).

### Стратегия

**Уведомление + кнопка** — APScheduler job находит старых пользователей с активной подпиской и отправляет им сообщение с кнопкой «Получить новый ключ».

### Алгоритм миграции одного пользователя

1. Найти активную Subscription в старой таблице (через devices → subscriptions)
2. Рассчитать новый `expire_at = subscription.end_date + 7 дней` (бонус за неудобство)
3. Взять `device_limit` из старой Device записи (обычно 1)
4. Вызвать `RemnawaveGateway.create_user(telegram_id, expire_at, device_limit)`
5. Сохранить `remnawave_uuid` и `subscription_url` в User
6. Создать новую Subscription и Payment запись в новом формате
7. Отправить пользователю subscription_url с инструкцией

### Что сказать пользователю

```
Уведомление:
«🔄 Мы обновили нашу систему VPN!
Ваша подписка активна до {дата} (+7 дней бонус).

Для подключения теперь используется happ —
один ключ для всех ваших устройств.

[🔑 Получить новый ключ]»

После нажатия:
«✅ Готово! Ваш ключ для happ:
{subscription_url}

📖 Как добавить ключ в happ — нажмите Инструкция»
```

### Job в APScheduler

- Разовый (oneshot) или с флагом «уже мигрирован» на User
- Запускается вручную администратором (через команду) или по расписанию
- Обрабатывает по одному пользователю с паузами (rate limit Telegram)

## Правила разработки

### Что нельзя делать

1. **Не создавать Device записи** в новом flow — только User → Subscription
2. **Не хранить device_limit** на нашей стороне как источник правды — он в Remnawave
3. **Не удалять Payment записи** каскадно — история платежей вечная
4. **Не использовать device_name** в renew flow — устройство ищем по telegram_id
5. **Не трогать старые device/subscription записи** — только читаем при миграции

### Что обязательно

1. При каждом `confirm_payment` — обновлять Remnawave (`create` или `update`)
2. Payment всегда содержит `user_telegram_id` (денормализация для истории)
3. При продлении — `end_date` считать от текущего конца подписки, а не от сегодня
4. `subscription_url` хранить на User, не пересоздавать при каждом обновлении

### Обратная совместимость

Старые пользователи (без `remnawave_uuid`) мигрируют при:
- Нажатии кнопки «Получить новый ключ» (через migration job)
- Либо автоматически при следующей оплате (confirm_payment уже это делает)
